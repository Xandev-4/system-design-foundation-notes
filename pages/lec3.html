Lecture 3

Scaling a Web App to Millions of Users

Progressive architecture upgrades — from single server to full distributed system

Overview

  • A practical exercise — take a tiny web app and progressively scale it from a few users to millions.
  • Each architectural upgrade is presented with pros and cons, simulating real backend engineering decisions.
  • Primary focus: the backend layer — it's almost always the first bottleneck.
  • Frontend is easy to scale (static files). Database is complex. Backend first, always.

1. Scalability (Review)

  • The ability to support an increasing workload without collapsing.
  • Dimension 1: ups and downs in daily/weekly/seasonal traffic — e.g., weekends busier, Black Friday spikes.
  • Dimension 2: long-term growth — next year you might have 2x the users.

Resource Utilization

  • Constant 70% utilization → probably fine.
  • Spikes to 90% → start thinking about scaling.
  • Never want to be at 100% — you need headroom for unexpected spikes.
  • Solution can be as simple as a stronger server, or as complex as a distributed system depending on scale.

2. Vertical vs. Horizontal Scaling (Review)

  • Vertical Scaling (Scale Up) — add more CPU/RAM to the existing server. Simple, but has a hardware ceiling.
  • Horizontal Scaling (Scale Out) — add more servers running in parallel. No hard ceiling, preferred at scale.
  • Most real-world solutions use a combination of both.
  • Horizontal scaling requires the backend to be structured to support it — stateless APIs work great; databases are harder.

Autoscaling

  • Horizontally scale up and down automatically based on current traffic.
  • Cloud providers handle this for you via virtualization.
  • Solves the daily/weekly traffic fluctuation problem.
  • This is what elasticity means — not the same as scalability.
Key Distinction Scalability = ability to handle growth over time. Elasticity = dynamically adjusting up AND down based on current load. Different concepts.

3. Typical 3-Tier App Structure

  • Frontend ↔ Backend API ↔ Database
  • Each tier may be scaled differently — don't treat them as one unit.
  • Frontend — usually static files, very easy to scale (serverless).
  • Backend — computation per request, first likely bottleneck.
  • Database — hardest to scale, tackle last.
Strategy Tackle backend first. Use caching to reduce DB load before ever having to scale the database directly.

Stage 1: Single Server (Monolith)

Everything — DB, backend, and frontend — lives on one single server.

  • Client makes request → hits one server → backend processes → fetches from DB → returns HTML.
  • Backend may use a template engine (Jinja2, Razor, Pug) to generate HTML dynamically.
  • Follows MVC pattern internally: Controller → Model → DB → View → Response.

Pros

  • Super simple for beginners — spin up one VM, run everything from it.
  • Simple to vertically scale — just add more RAM or CPU.

Cons

  • Cannot scale sections independently — if DB needs more RAM but backend doesn't, you can't split the upgrade.
  • No separation of concerns — everything tightly coupled, hard for larger teams to work in.
  • Single Point of Failure (SPOF) — if the server goes down, EVERYTHING goes down.
  • DDoS vulnerability — one server to flood with traffic and your app crashes.
  • Limited to vertical scaling — diminishing returns at large scale.
Best For Small personal projects with very few users. Not suitable for production at scale.

Stage 2: Three-Tier Architecture

Split the monolith into three dedicated servers.

Frontend Server

Standalone React app — just HTML, CSS, JS. Hosted on a serverless static file service.

AWS → S3 / Vercel / Netlify

Backend Server

API layer handling computation per request. First bottleneck to tackle.

AWS → EC2

Database Server

Dedicated DB instance. Scale independently from backend.

AWS → RDS

Frontend in 3-Tier

  • No longer uses template engines — now a standalone React app.
  • Hosted on a serverless static file service — no compute needed server-side.
  • JS in browser makes API calls to the backend.
  • Not the bottleneck — you only pay for usage, no server provisioning needed.

Pros

  • Very common — well-documented, easy to find support.
  • Independent scaling — scale DB, backend, or frontend separately.
  • Easier to find bottlenecks — each layer has its own resource metrics.

Cons

  • Requires an API layer — now you must design and maintain a proper backend API.
  • CORS complexity — frontend and backend on different domains → configure cross-origin requests.
  • Networking setup — backend and DB must be configured to communicate securely.
  • Still vertical scaling only for backend and DB — no horizontal scaling yet.
  • Still has SPOF at the backend level — one backend server is still a single point of failure.

Stage 3: Load Balancing + Horizontal Scaling

The next logical step: introduce horizontal scaling for the backend.

The Problem

  • One backend server = single point of failure.
  • If it goes down → entire app breaks.
  • Can't handle unlimited traffic growth with one node.

The Solution

  • Add multiple backend servers (nodes) running the same API code.
  • A Load Balancer sits in front and distributes incoming requests across all servers.
  • Frontend only knows one endpoint (e.g., api.mysite.com) → points to load balancer's IP.
  • If one server goes down, load balancer redirects traffic to healthy nodes automatically.
  • Load balancer also balances traffic evenly — no single server gets overloaded.

Why APIs Scale Horizontally So Well

  • APIs are typically stateless — they don't store user session data locally.
  • Each request is self-contained, so any node can handle any request.
  • This is the key property that makes horizontal scaling easy for backends.

Pros

  • Removes SPOF from backend → High Availability.
  • Failover: if a node dies, traffic automatically reroutes → Fault Tolerant.
  • Horizontally scalable — add/remove servers as needed.
  • Cheaper at scale than vertical scaling — no diminishing returns.
  • Supports autoscaling — provision/deprovision nodes dynamically based on traffic spikes.

Cons

  • Must add and configure a load balancing service.
  • No caching yet — every request hits the DB fresh, even if 80% ask for the same data.

Stage 4: CDN (Content Delivery Network)

The Problem

  • If your server is in the US and user is in India, every request travels a massive distance — adding latency.

The Solution

  • A CDN distributes cached copies of static content (HTML, CSS, JS, images, videos) across multiple edge nodes worldwide.
  • User's request goes to the nearest CDN edge node — not all the way to the origin server.
  • Popular CDN: Cloudflare.

How It Works

  • User makes request → DNS resolves to nearest CDN edge node.
  • Cache hit → edge node responds directly (super fast).
  • Cache miss → edge node fetches from origin server, caches it, responds to user.

What CDNs Cache

  • Static content: HTML, CSS, JavaScript, images, videos, downloadable files.
  • Some API responses that change infrequently — but CDN is primarily for static files.
  • A separate caching layer (Redis) handles dynamic API data.

Pros

  • Reduced latency — content served from nearest edge node, not main server.
  • Reduced server load — requests never reach origin server if cached.
  • Easy global reach — acts as a "cheat code" to make your app feel global without multi-region deployment.
  • DDoS protection — many CDNs (Cloudflare) include built-in attack mitigation.
  • Cost savings — especially for apps heavy on images, videos, and large file downloads.
  • SEO benefits — faster page loads improve search rankings.

Cons

  • Increased complexity — you must understand how TTL (Time-To-Live) works.
  • Stale data risk — cached content may be outdated until TTL expires or you manually purge it.
  • Cache invalidation — need a strategy for when files change.
  • Cache hit ratio — need to monitor what % of requests are served from cache vs. origin.

Stage 5: Caching Layer (Redis)

The Problem

  • Even with horizontal scaling, every API request hits the database — even for data that rarely changes.
  • At millions of users, this is a massive load on the DB.

The Solution

  • Add an in-memory cache (Redis) between the backend and database.
  • Cache hit — data found in cache → return immediately, no DB query needed.
  • Cache miss — data not in cache → fetch from DB → store in cache → return to client.
  • First request still goes to DB, but all subsequent requests for same data are served from RAM — orders of magnitude faster.

The 80/20 Rule

  • In most apps, 80% of requests ask for 20% of the data.
  • Cache just that 20% in RAM → reduce DB queries by up to 80%.
  • Also helps with complex queries — a 200ms JOIN query cached = instant on all subsequent hits.

Cache Invalidation Strategies

  • Update both DB and cache together — most reliable, keeps them in sync.
  • Set a TTL (Time-To-Live) — cache expires after X seconds; briefly stale but auto-self-heals.
  • Clear the cache on write — cache is deleted; next read is a cache miss, fetching fresh data from DB.
Cache is Volatile! Redis stores data in RAM — if the server restarts, the cache is cleared. This is fine because the DB is the source of truth. It's just a temporary performance hit on restart (cold start).

Scaling the Cache

  • You can horizontally scale Redis (Redis Cluster) before needing to scale the DB.
  • Often cheaper and easier than scaling the database.
  • Scale the cache first. Only scale the DB when you truly have to.

Pros

  • Drastically reduces DB compute load.
  • Improves response speed / reduces latency.
  • Likely cheaper overall — less DB compute needed.
  • Can cache results of complex queries, not just simple lookups.

Cons

  • Increases architectural complexity.
  • Cache invalidation is a notoriously hard problem.
  • TTL management — balance freshness vs. performance.
  • Data can be temporarily out of sync between cache and DB.
  • Cache is volatile — restarting clears it (cold start performance hit).

Stage 6: Database Layer Scaling

After optimizing the backend, the database becomes the next bottleneck.

Primary/Replica Architecture

  • All writes go to the primary (write) node.
  • All reads are distributed across multiple replica (read) nodes.
  • Most apps are read-heavy (~80% reads) — this setup scales reads efficiently.
  • A standby write node can be on standby for failover.

NoSQL vs SQL for Horizontal Scaling

  • Some NoSQL databases (Cassandra, DynamoDB, MongoDB) are designed from the ground up for horizontal scaling — much easier to shard.
  • SQL/relational databases can be horizontally scaled but it's more complex — sharding, managing transactions across nodes.
  • If you know your app will be at massive scale, NoSQL may be a better fit — but it's a big design decision.

Full Architecture at Scale

Request Flow User types URL → DNS resolves to CDN edge or frontend → Browser loads static files (from CDN if cached) → JS makes API call → hits Load Balancer → routed to one Backend Node → checks Redis cache → Cache hit: return immediately / Cache miss: query DB, store in cache, return → response travels back to user

All Layers Stack

  • CDN — serves static files from nearest edge node worldwide.
  • Frontend — serverless static hosting (S3/Vercel), zero compute server-side.
  • Load Balancer — single endpoint for frontend, distributes to backend nodes.
  • Backend Nodes — stateless, horizontally scaled API servers.
  • Redis Cache — in-memory layer, absorbs 80% of DB queries.
  • Database Cluster — write node + read replicas + standby failover node.

Evolution Summary

  • Stage 1 — Single Server: Simple to set up. SPOF, no independent scaling, vertical only.
  • Stage 2 — 3-Tier: Independent scaling per layer. Still vertical only, still SPOF at backend.
  • Stage 3 — Load Balancer + Horizontal Scaling: Removes backend SPOF, High Availability. No caching, every request hits DB.
  • Stage 4 — CDN: Reduces latency globally, reduces static file load. Stale cache risk, added complexity.
  • Stage 5 — Redis Caching: Drastically reduces DB load, faster responses. Cache invalidation complexity.
  • Stage 6 — Database Clustering: Scales reads, removes DB SPOF. Most complex to implement.

Key Mental Models

  • Stateless backends scale easily horizontally — session state should live in DB or cache, never in-memory on the server.
  • Scale DB last — use caching to reduce DB load first. Only scale the DB when you truly have to.
  • Vertical scaling has diminishing returns — at massive scale, horizontal is always cheaper.
  • CDN = static content optimization. Redis = dynamic content optimization. They solve different problems.
  • Cache invalidation is hard — always have a clear strategy for keeping cache and DB in sync.
  • More layers = more complexity = more things that can fail — only add layers when justified by actual scale needs.
