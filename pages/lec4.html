Lecture 4

Horizontal and Vertical Scaling

Deep dive — when to use each, tradeoffs, and a full decision framework

Overview

  • A deep dive into horizontal vs. vertical scaling — building on the previous video.
  • Goal: understand when and why to choose each, with focus on the API layer first, then databases and frontend.
  • The honest answer is always "it depends" — but this video gives you a real framework for making that decision.

1. In-House vs. Cloud Environments

Before anything else — where are your servers running? This changes the scaling math significantly.

Cloud (Virtualized)

  • You don't touch physical hardware — a cloud provider (AWS, GCP, Azure) handles it.
  • Behind the scenes, they're not physically swapping hardware — they're re-allocating virtual resources from a shared pool.
  • You just change your instance size and the change is instant.
  • No space constraints — want 20 servers? Done. Scaling is significantly easier and faster.

On-Premises (In-House)

  • You physically own the servers, possibly in a data center you can visit.
  • Space constraints — you can't endlessly add servers; they take up physical room.
  • Replacing individual hardware components (e.g., swapping in more RAM) is a real physical operation.
  • Diminishing returns on hardware cost are very real here.
Context For most of this video, Caleb teaches from the cloud perspective. On-prem context is only flagged where it meaningfully differs.

2. Quick Definitions (Review)

  • Horizontal Scaling (Scale Out) — add more servers running in parallel. One puny server → five puny servers working together as one unit.
  • Vertical Scaling (Scale Up) — upgrade the existing server (more CPU, RAM, etc.). One puny server → one much beefier server.
  • The user never notices the difference — they just see one service endpoint either way.
Important Caveat There is no universal best answer. The right choice depends on the type of service — API, database, frontend, cache, queue. The video builds the framework starting with the easiest case: the stateless backend API.

3. Stateless Services & APIs

Why APIs Scale So Easily

  • The backend API layer is the easiest service to scale because it is typically stateless.
  • No user session data stored in-memory on the server.
  • No database instance living inside the backend server.
  • Each request is completely self-contained — any server can handle any request.

The Danger of a Stateful API Server

  • If your API server also hosts the database locally, adding a second API server creates two separate database instances → data gets out of sync.
  • A write to Server 1's DB won't be reflected on a read from Server 2's DB.
  • Fix: move the database to a separate, dedicated server. Both API nodes share one external database → API is now truly stateless.
Architecture WRONG: [API + DB] on each node. RIGHT: [API Node 1] + [API Node 2] + [API Node 3] → all pointing to one [Shared DB Server].

4. What Can You Vertically Scale?

When vertically scaling a server, there are four core hardware components to look at:

  • CPU — processing power; how fast requests are computed.
  • RAM — in-memory data storage; affects caching, concurrent connections.
  • Disk — storage size + speed (SSD vs HDD); more relevant for databases.
  • Network — bandwidth; how much data can move in/out per second.
Note For stateless APIs, disk is rarely the bottleneck since nothing is stored. CPU and RAM are the primary concerns.

Balancing Hardware Utilization

  • Goal: have all four components roughly balanced — close to the same % utilization.
  • If wildly unbalanced, you're overpaying for something or under-provisioning something else.

Example Scenario

  • CPU: 90% | RAM: 20% | Disk: 20% | Network: 20%
  • Option A — Double the whole server: CPU drops to 45%, but RAM/Disk/Network drop to 10% — wastefully underused, double the money.
  • Option B — Just upgrade the CPU (2x): CPU: 45% | everything else stays. Much more cost-efficient — only upgrade what's needed.
Target Utilization Aim for ~75% peak utilization across all components. Gives you headroom for spikes without wasting money. If traffic is steady and predictable, you can push closer to 90%. If traffic spikes wildly, keep more headroom.

5. Steps to Scale an API Layer

The recommended step-by-step process for scaling a backend API:

Step 1 — Vertical Scale to a Reasonable Starting Point

  • Adjust the instance type (CPU, RAM) to match the workload.
  • Don't over-provision — find an instance where utilization is roughly balanced and reasonable.
  • Goal: not running at 20% waste, not pushing 95% and risking crashes.
  • This is NOT about making a supercomputer — just get the right-sized instance.

Step 2 — Horizontally Scale to at Least 2 Nodes

  • This is non-negotiable for production.
  • Eliminates the Single Point of Failure (SPOF) on the backend.
  • Provides failover — if one node dies, the other takes all traffic.
  • Enables High Availability — users don't experience downtime during node failures.
  • Requires a Load Balancer in front — easy to set up in cloud environments.
  • Once set up, adding more nodes is trivial — just attach them to the load balancer.

Step 3 — Choose: Continue Vertical or Horizontal?

  • Once you have the 2-node setup, you have flexibility.
  • General rule: horizontal scaling is preferred for stateless APIs — no hardware ceiling.
  • But don't use the cheapest possible instances — use decent instances so you don't need a hundred of them.
  • Let cost guide the decision (see: Diminishing Returns section).
Summary Step 1: Fine-tune instance size (vertical). Step 2: Add a second node + load balancer (horizontal, for HA). Step 3: Scale further via horizontal (preferred) or vertical (if still cost-effective).

6. The Tradeoff Framework

Diminishing Returns (Vertical Scaling)

  • As you spend more money on a single server, you get less performance improvement per dollar.
  • The cost curve for hardware is exponential — doubling performance costs more than double the money.
  • Real example: going from a 16-core CPU to a 32-core CPU gives ~2x performance, but the 32-core chip costs more than 2x the 16-core chip.

Laptop Analogy

  • $800 laptop → 64 GB storage (basically useless).
  • $1000 laptop → 1 TB storage (20x improvement for $200 more).
  • $1200 laptop → 2 TB storage (2x improvement for another $200).
  • $2400 laptop → maybe 3–4 TB (not 2x the performance for 2x the money).

The Cost Crossover Point

  • Vertical scaling is more cost-effective at small scale (before the crossover).
  • Horizontal scaling is more cost-effective at large scale (after the crossover).
  • There's an initial flat cost to every server (manufacturing, networking, base cost) regardless of how powerful it is — that's the initial steep part of the horizontal curve.
  • The switch point: when the cost of vertically upgrading hardware exceeds the cost of adding another server.

Virtualized Cloud Caveat

  • In cloud environments, cloud providers aren't actually swapping physical chips — they're reallocating virtual resources from a shared pool.
  • Going from medium to large instance may be close to 2x cost for 2x performance — more linear than physical hardware.
  • Diminishing returns are more pronounced on physical/on-prem hardware.
  • Still useful to understand the theory — just know real-world cloud numbers may behave more linearly.

7. Scaling the Frontend

  • Frontend scaling is largely not a concern in modern architectures.
  • Host as a static site (HTML, CSS, JavaScript) on a serverless service.
  • AWS: S3 static hosting; alternatives: Vercel, Netlify, Cloudflare Pages.
  • You pay only for usage — no server provisioning, no vertical/horizontal scaling decisions.
  • The cloud provider handles all scaling behind the scenes automatically.
  • Works well because frontend files are simple assets with no server-side computation.
TL;DR for Frontend Use serverless static hosting and forget about it. Scaling is not your problem.

8. Scaling the Database

The database is the hardest layer to scale. Unlike stateless APIs, databases are stateful — you can't just spin up identical nodes without coordination.

Strategy 1: Vertical Scaling First (Relational / SQL)

  • For relational databases (PostgreSQL, MySQL), vertical scaling first is the recommended default.
  • Scale up to a very powerful database server — cloud providers offer extremely high-spec instances.
  • You'd be surprised how far a single beefy database server can take you.
  • Avoids the enormous complexity of distributed database coordination.
  • Downside: single point of failure.

Eliminating the DB SPOF

  • Standby Database — a second DB instance synchronized with the primary but not actively serving traffic. Only activates if the primary fails. Provides failover + High Availability without the complexity of active clustering.
  • Read Replicas — copy data from primary to one or more read-only nodes. All writes → primary. All reads → read replicas. Can distribute replicas across geographic regions to reduce latency. Scale read nodes independently from write node.
Good For Apps with a high read-to-write ratio (e.g., social media feeds — one post per day, thousands of reads per day). Limitation: still only one write location → adds latency for writes from geographically distant clients.

Strategy 2: Multi-Master Clustering (Multiple Write Nodes)

  • When a single write node becomes the bottleneck, add multiple write nodes.
  • Massively increases complexity — write conflicts, ordering guarantees, consistency across all write nodes.
  • Caleb's advice: don't add multi-master unless you actually need it. Premature optimization is a real trap.
  • If you're building the world's largest social network — go for it. Otherwise — stick with single write node + read replicas.
  • Dedicated videos on database clustering planned in the series.

Strategy 3: NoSQL for Write-Heavy Systems

  • NoSQL databases (MongoDB, Cassandra, DynamoDB) are built from the ground up for horizontal scaling and multi-write scenarios.
  • Natively designed for distributed writes — less rigidity in schema = easier to shard and distribute.
  • Better choice when your app is write-heavy (e.g., messaging apps, real-time event streaming).
  • You get multi-write horizontal scaling benefits out of the box vs. bolting it onto a relational DB.

10. Decision Framework Summary

For Stateless APIs (Backend)

  • Tune instance size (vertical) so resources are balanced and ~75% utilized at peak.
  • Always add at least 2 nodes + load balancer (horizontal) for HA/failover — non-negotiable for production.
  • After step 2, prefer horizontal scaling — cheaper at scale, no hardware ceiling.

For Databases (Relational / SQL)

  • Vertical scale first — avoid distributed complexity as long as possible.
  • Add a standby replica to eliminate SPOF.
  • Add read replicas for read-heavy workloads.
  • Only introduce multi-master if truly needed — you'll know when you need it.

For Databases (NoSQL)

  • Use NoSQL when you need native horizontal scaling or are write-heavy.
  • Horizontal scaling is built in — much simpler than sharding relational DBs.

For Frontend

  • Use serverless static hosting — scaling is handled for you, no decisions needed.

Key Principles to Remember

  • "It depends" is always the correct answer — but now you have a framework for why it depends.
  • Don't over-engineer — only add complexity (horizontal DB clustering, multi-master) when you've exhausted simpler options.
  • Premature optimization is the enemy — scale when you need to, not when you think you might someday need to.
  • In the cloud, vertical scaling is more accessible — virtualization makes instance upgrades cheap and instant.
  • On-prem/physical hardware = stronger case for horizontal — hardware costs are truly exponential.
  • The crossover point (where horizontal becomes cheaper than vertical) is the natural trigger to switch strategies.
  • Stateless = scales horizontally with ease. Stateful = harder to scale horizontally.
