Lecture 5

Scaling to Global Levels

Multi-region architecture, GeoDNS, CDN edge workers, and data localization laws

Overview

  • Builds on horizontal scaling and zooms out to answer: where are your servers physically located?
  • 10 servers in the same building = one power outage wipes everything.
  • Covers 4 progressive levels of geographic distribution — from a single data center to globally distributed infrastructure across continents.

1. The Core Question: Server Proximity

When you horizontally scale, you need to decide how close together those servers are. Each level adds more High Availability, more latency improvement for distributed users, and more complexity and cost.

  • Level 1 — Same Data Center
  • Level 2 — Same Region, Different Availability Zones
  • Level 3 — Different Regions (within a continent)
  • Level 4 — Different Continents

2. Level 1 — Same Data Center

A data center = a building (or campus) in a single physical location, housing servers run by a company or cloud provider.

What It Improves

  • ✅ Scalability — more servers = more throughput.
  • ✅ High Availability (partial) — if one server crashes, the other takes traffic.

What It Does NOT Improve

  • ❌ Latency — both servers are the same physical distance from any user; routing to either one takes the same time.
  • ❌ True fault tolerance — a power outage or physical disaster at the data center takes both servers down simultaneously.
Best For Simple scaling where you just need more compute, not geographic resilience.

3. Level 2 — Same Region, Multiple AZs

What is an AWS Region?

  • A region is a named geographic grouping of data centers.
  • AWS has 8 North American regions (e.g., us-east-1, us-east-2 Ohio, us-west-1, us-west-2, etc.).
  • Pick the region closest to your user base — e.g., most users in California → pick a US West region.

What is an Availability Zone (AZ)?

  • Within a single region, AWS has multiple Availability Zones (e.g., us-east-2a, us-east-2b, us-east-2c).
  • Each AZ is a logically separated data center — different buildings, different power grids.
  • AZs are connected with high-speed, low-latency links for efficient inter-AZ communication.
  • Both AZs are still in the same broad geographic area (e.g., both in Ohio).

What It Improves

  • ✅ Scalability — still have multiple servers.
  • ✅ High Availability (stronger) — if an entire AZ goes down, servers in other AZs stay up.

What It Does NOT Improve

  • ❌ Latency — servers are still geographically close; a request from California hits roughly the same latency to either AZ in Ohio.
Recommendation Do this as a baseline for any production system. Low complexity, low additional cost, massively improves resilience against data-center-level failures.

4. Level 3 — Multiple Regions (Within a Continent)

  • Servers in two different AWS regions — e.g., us-east-2 (Ohio) and us-west-2 (Oregon).
  • Now you have servers on the East Coast and West Coast of the US.
  • You can't blindly round-robin traffic across regions — that defeats the whole point.
  • If a user in California gets routed to an East Coast server, you've increased their latency for no benefit.
  • Solution: Geolocation-Based DNS Routing (GeoDNS).

5. Geolocation Routing (GeoDNS)

GeoDNS directs clients to the nearest region based on their geographic location, at the DNS resolution level.

How It Works

  • Client makes a request → DNS lookup.
  • GeoDNS determines client's approximate location (by IP).
  • DNS returns the IP/endpoint of the closest region.
  • Client connects to that region's load balancer.
  • Region's load balancer routes to one of that region's servers.
AWS Service Route 53 supports geolocation and latency-based routing policies. Client in California → US West Region. Client in New York → US East Region. Client in London → EU Region.

Architecture of Each Region

  • Each region acts as an independent scaled unit with its own Load Balancer.
  • Multiple backend servers behind the LB, split across multiple AZs (Level 2 within Level 3).
  • Regions can be scaled independently — e.g., US East gets 75% traffic → give it 3 servers; US West gets 25% → give it 2 servers.

6. The Database Problem in Multi-Region

The Core Issue

  • If you have a single central database in one region, you've created a major problem.
  • User in California → routed to US West API server (fast ✅) → must query database in US East (slow ❌).
  • Net result: same or worse total latency than just sending the user to US East directly.
The Latency Math Client → US West API: 100ms + US West API → US East DB: 200ms = 300ms total. vs. Client → US East API: 200ms + US East API → US East DB: 100ms = 300ms total. Spreading the API layer is pointless if the DB is still far from the API servers.

Solution 1 — Multi-Master Database

  • Have a database instance in each region that can accept writes.
  • West Coast API → West Coast DB (writes & reads). East Coast API → East Coast DB (writes & reads). DBs sync with each other.
  • Benefit: writes and reads are low-latency everywhere.
  • Challenge: write conflicts, consistency management — covered in future DB videos.
  • Example service: DynamoDB Global Tables (handles multi-region write/read natively).

Solution 2 — Read Replicas Per Region

  • One primary write node (single region) + read replicas deployed in each region.
  • All reads served locally (low latency). All writes go to central primary (still some latency for writes).
  • Best when: app is read-heavy (social media, news feeds, etc.).
  • Simpler than multi-master but doesn't solve write latency.

Solution 3 — Caching Per Region

  • Deploy a Redis cache in each region.
  • API reads from local cache first → DB only on cache miss.
  • Dramatically reduces how often the far-away DB is hit.
  • Best for: data that doesn't change frequently. Can even cache entire API responses (HTTP caching).
  • Great incremental improvement, easy to implement.

7. Multi-Region Benefits & Challenges

Benefits

  • Lower Latency — requests served from geographically nearest server.
  • Higher Availability — a regional failure (datacenter fire, network outage) doesn't bring down the whole service.
  • Independent Scalability — scale each region's compute based on that region's traffic share.
  • Disaster Recovery — can failover traffic between regions if one goes dark.

Challenges

  • Complexity — GeoDNS setup, inter-region communication, database sync.
  • Data Transfer Costs — cloud providers charge for data movement between regions — budget for this.
  • Service Coordination — any service calling another service in a different region adds cross-region latency.
  • Still not truly global — multiple US regions doesn't help users in Asia or Europe.

8. Level 4 — Different Continents

Same principles as multi-region, but regions are now in different continents — North America, Europe, Asia-Pacific, etc. Latency improvement becomes very significant — a cross-Atlantic request adds hundreds of milliseconds.

Data Localization Laws

Once you deploy servers in other countries, you enter legal and compliance territory that varies massively by country and data type.

  • United States — CUI (Controlled Unclassified Information) — sensitive government-related data cannot be stored on servers outside the US. AWS GovCloud exists specifically for these compliance needs.
  • Europe — GDPR (General Data Protection Regulation) — health records, personal data of EU citizens may be required to remain physically within the EU. Non-compliance = fines up to 4% of global revenue.
  • China — strong data sovereignty laws. Apps operating in China often need servers physically within China and compliance with local regulations.
Caleb's Note This is not his area of expertise. Do deep research specific to your app's data type and target countries before deploying cross-continent. The consequences of getting this wrong are severe.

AWS GovCloud

  • Specialized AWS regions for US government/compliance requirements.
  • Higher security standards, restricted access.
  • Used for classified data handling, defense applications.

9. Multi-Region Failover

  • If an entire region goes down, automatically redirect all its traffic to another region.
  • Configured at the GeoDNS/Route 53 level with health checks + routing policies.
  • West Coast region dies → Route 53 detects health check failure → reroutes all West Coast clients to East Coast.
  • Users experience higher latency temporarily but the service stays up.
  • No manual intervention needed if set up correctly.
  • Distinct from normal GeoDNS (which routes by location) — failover routing kicks in only when a health check fails.

10. CDN — The Global Shortcut

A CDN gives you many benefits of global infrastructure without the complexity of managing multi-region backend servers.

How CDNs Work in This Context

  • CDN providers (Cloudflare, AWS CloudFront) operate edge servers spread across every continent.
  • Static files (HTML, CSS, JS, images, videos) are cached on the nearest edge node.
  • A user in Tokyo hits a Tokyo edge node — not your US origin server.
  • On cache miss → edge node fetches from origin, caches it, responds to user.

CDN vs Full Multi-Region Backend

  • CDN — best for static files and infrequent-change API responses. Low complexity, low cost, no server-side computation.
  • Multi-Region Servers — best for dynamic API responses and real-time data. High complexity, high cost, full computation at each edge.

Edge Workers (Advanced CDN Concept)

  • Some CDN providers (e.g., Cloudflare Workers) allow running actual server-side computation at edge nodes.
  • Not just caching — these are mini-servers at every edge location.
  • Essentially distributes your API layer globally without managing regional infrastructure.
  • Very powerful for low-latency global APIs — worth researching as an alternative to full multi-region.
When to Add a CDN Add a CDN early — almost first thing after your initial deployment. Benefits are immediate: instant global performance improvement, DDoS protection (built into Cloudflare), SEO improvement, reduced origin server load.

11. Full Global Request Flow

  • Client types URL → DNS lookup.
  • GeoDNS (Route 53) → routes to nearest region.
  • CDN Edge Node → static files served here if cached. Dynamic API requests pass through.
  • Regional Load Balancer → routes to available server in healthiest AZ.
  • Backend Server → checks local cache first.
  • Regional Redis Cache → cache hit: return immediately. Cache miss: hit DB.
  • Read Replica (local) or Primary Write DB (possibly remote) → data returned.
  • Response travels back up the chain to the client.

12. All 4 Levels Summary

Level 1

Same Data Center. Partial HA (node failures only). No latency improvement.

Complexity: Low

Level 2

Same Region, Multi-AZ. Strong HA (AZ failures). No latency improvement.

Complexity: Low

Level 3

Multi-Region, same continent. Very strong HA. Moderate latency improvement.

Complexity: Medium-High

Level 4 Multi-Continent. Maximum HA. Maximum latency improvement for global users. Very High complexity + legal compliance requirements per country.

Key Principles to Remember

  • Geolocation routing ≠ load balancing — they solve different problems (location vs. load distribution).
  • Multi-region API is useless without addressing the database — the DB is the bottleneck for cross-region latency.
  • Read replicas are the easiest first DB fix for multi-region — read-heavy apps benefit most.
  • CDN first — get the low-hanging fruit before building complex multi-region architecture.
  • Edge Workers = the modern "cheat code" for globally distributed compute without managing full regional infrastructure.
  • Legal compliance is non-negotiable — always research data laws before deploying in a new country.
  • Multi-region failover = automatic disaster recovery at the geographic level.
