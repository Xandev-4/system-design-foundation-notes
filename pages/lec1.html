Lecture 1

Introduction to System Design

MVC and Three-Tier Architecture

Key Vocabulary

  • Design Pattern — how you organize code within a single project/application.
  • Architecture — how different systems (servers, databases, services) interact to build a larger system.
  • System Design — the process of solving a specific large-scale problem end-to-end, usually resulting in an architecture.
TL;DR Design Pattern = code organization. Architecture = system organization. System Design = the process of getting to an architecture.

What is System Design?

  • The process of defining the architecture, components, and data flow of a system to meet specific requirements.
  • Happens before coding — it's the blueprint of your application.
  • Covers both high-level (big picture, services, databases) and low-level (classes, functions, algorithms) design.
  • Goal: build systems that are scalable, reliable, maintainable, and efficient.

Why System Design Matters

  • Bad design early = technical debt that compounds over time.
  • A well-designed system handles growth — more users, more data — without breaking.
  • It's the difference between a project that works and a product that scales.

MVC — Model View Controller

A design pattern that splits your application code into three logical sections. All of it still lives in one project/repo.

  • Model — talks to the database; responsible for fetching/storing data.
  • View — responsible for presenting data (generates HTML + CSS).
  • Controller — the coordinator; receives the request, talks to the model, passes data to the view.
Request Flow User Request → Router → Controller → Model → Database → Controller → View → HTML Response → User

Why MVC?

  • Separation of concerns — each layer has one job.
  • Easier to test, maintain, and scale individual parts.
  • Multiple Views can use the same Model (e.g. web + mobile app).

MVC in the Real World

  • Django (Python) — follows MVC (calls it MTV: Model-Template-View)
  • Express.js — you manually implement MVC pattern
  • Spring Boot (Java) — built around MVC

Three-Tier Architecture

A physical/deployment model that splits the system into three distinct layers. Each tier runs as its own separate service/server.

Tier 1 — Presentation

The frontend / UI. What the user interacts with directly.

e.g. React app, mobile app

AWS → S3

Tier 2 — Application

The backend / business logic. Processes requests, applies rules.

e.g. Node.js / Express server

AWS → EC2

Tier 3 — Data

The database layer. Stores and retrieves data.

e.g. PostgreSQL, MongoDB

AWS → RDS

Monolith → 3-Tier: What Actually Changes?

  • Split the frontend into its own application (e.g. React).
  • Change the database connection string to point to a separate DB server.
  • That's literally it code-wise — the big change is architectural.

Why Separate into 3 Tiers?

  • Independent scalability — scale only the layer that needs it.
  • Need more backend compute? Spin up 3 backend servers without touching frontend or DB.
  • Need more DB capacity? Scale the database layer on its own.

New Challenges Introduced

  • Load balancing — which backend server does the frontend talk to?
  • Network communication between services adds latency.
  • More moving parts = more complexity to manage.
Caleb's Advice Start with 3-tier from day one. Running everything on a single server is uncommon for any real app — skip the full monolith phase.

MVC vs Three-Tier

  • MVC is a code organization pattern — how you structure your code.
  • Three-Tier is a deployment/infrastructure pattern — how your system is physically separated.
  • They coexist — your backend tier can internally follow MVC.

Quick Recap

  • Design Pattern → organizes code (e.g. MVC)
  • Architecture → organizes systems (e.g. 3-Tier)
  • System Design → the process of solving a large-scale problem → ends with an architecture
  • MVC flow: Router → Controller → Model → DB → Controller → View → User
  • 3-Tier: Frontend Server | Backend Server | Database Server
  • Monolith: all of the above on one single server
