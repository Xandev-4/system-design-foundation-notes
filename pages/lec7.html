Lecture 7

Replication and Sharding

Database scaling deep dive — from single node to multi-master and sharded clusters

Overview

  • A focused deep dive into scaling a single DB into a multi-node cluster using replication and sharding.
  • Framed around relational databases first (harder to distribute), then extends to NoSQL.
  • Covers the full progression: single node → standby → read replicas → multi-master → sharding.

1. Start with a Single Relational DB

Default First Step: Vertical Scale

  • Vertically scale the single database instance first — more CPU, RAM, faster disk.
  • For relational DBs, you want a powerful single server — not a bunch of average ones.
  • A single well-provisioned server is adequate for many production use cases.
  • Don't jump to multi-node complexity until you've exhausted what a single node can do.

Reduce Load Before Scaling

  • Redis Caching — frequently requested data served from memory → fewer DB queries.
  • Specialized NoSQL DBs — offload specific workloads (search, sessions, analytics) to purpose-built databases.
  • Data Warehousing — move analytics queries to a separate OLAP system.
Key Point These techniques can extend the life of your single database significantly — and are cheaper than multi-node infrastructure. Reduce demand before you scale.

2. Specialized NoSQL Databases

NoSQL is not a single type — it's a category of specialized databases. Use one or more alongside your main relational DB for specific workloads.

  • Redis (In-Memory Cache) — temporary data, sessions, caching frequent reads.
  • Elasticsearch (Search Engine DB) — full-text search, complex querying. Offloads search from main DB.
  • IndexedDB (Client-Side DB) — browser-local storage for games, settings, offline-first apps. Never hits the server.
Important These are NOT replacements for your main DB. They're targeted tools for specific jobs. Your main relational DB stays as the source of truth — these sit alongside it and handle specialized loads.

3. OLTP vs. OLAP

OLTP

Online Transactional Processing. Day-to-day app operations — CRUD.

Creating a user, placing an order

PostgreSQL, MySQL

OLAP

Online Analytical Processing. Analytics, reporting, aggregations.

"Show me all sales in Q3 by region"

Redshift, BigQuery, Snowflake

Why Split Them?

Analytics queries are computationally expensive — full table scans, big JOINs. Running these on your OLTP DB competes with regular app traffic.

Move analytics to a dedicated data warehouse

  • Main DB is now free to focus purely on transactional work once analytics are offloaded.

4. High Availability — The Standby Node

After optimizing the single node, the first multi-node concern is not performance — it's removing the Single Point of Failure.

Standby Node (Hot Failover)

  • A second database instance that is a live copy of the primary.
  • Data is replicated from primary → standby in real-time.
  • The standby is not active — receives no read or write traffic under normal operation.
  • If the primary fails → traffic is automatically redirected to the standby.
  • Zero (or near-zero) downtime during failover — users don't notice.
Flow Normal: Primary (R/W) active → Standby silent. After Failure: Primary goes down → Standby becomes new Primary (R/W).

How Replication Works (Technical)

  • Internally, databases use a Write-Ahead Log (WAL) — every operation is logged.
  • This log is used to replay the same operations on the standby.
  • In practice: every incoming write is written to both the primary and standby.
  • This happens automatically — your application code doesn't need to write twice.

Standby Node vs. Data Backup

  • Standby Node — a live running DB instance. Near-instant failover (already running). Purpose: High Availability.
  • Data Backup — just a copy of data files. Slow to restore. Purpose: long-term disaster recovery.
  • Both are important and serve different purposes — one is not a substitute for the other.

5. Why Replication Doesn't Help Write Performance

This is a common misconception — important to understand clearly.

Key Fact Adding a standby node does NOT increase write throughput.
  • Every write to the primary is also replicated to the standby → 1 write operation becomes 2 write operations.
  • If each node handles 1,000 writes/sec → total system capacity = still ~1,000 writes/sec.
  • The extra compute is consumed by replication itself — it cancels out.

Even in Multi-Master

  • Writing to Node A → replicates to Node B (2 writes). Writing to Node B → replicates to Node A (2 writes).
  • Effective write throughput is cut in half — you're back to ~1,000 writes/sec total.
  • 1:1 full replication does not scale writes.
  • To actually increase write capacity, you need sharding (partial replication) — covered next.

6. Read Replicas — Scaling Reads

While replication doesn't help writes, it unlocks horizontal scaling for reads.

  • All writes → primary write node only.
  • All reads → distributed across read replica nodes.
  • Read replicas receive data via replication from the primary.

Benefits

  • Scale reads independently — add more read replicas without touching the write node.
  • Tune compute independently — write node gets write-optimized specs; read nodes tuned for query performance.
  • Geographic distribution — place read replicas in different regions (US, Europe, Asia) to reduce read latency for global users.
  • Read replicas can also have standbys — if a replica goes down, its standby takes over (lower priority than primary standby).
Failover Note If a read replica fails and has no standby, traffic just routes to another read replica — no downtime. That's why read replica standbys are lower priority than the primary write node standby.

Eventual Consistency in Read Replicas

  • Read replicas are eventually consistent by default — replication has a small delay (milliseconds to seconds).
  • Write a blog post → it's on write node instantly. Read immediately from a replica → might not have it yet. Wait a moment → all nodes sync.
  • Fine for most apps: social media, blogs, catalogs, feeds.
  • Not fine for: banking (stale account balance after a transaction), inventory (showing "in stock" after item sold).
  • Fix: configure synchronous replication — write not confirmed until read replica also confirms. Slower, but every read guaranteed up-to-date.

7. Distributing Read Traffic Across Replicas

When you have multiple read replicas, how do you decide which one handles each read?

  • Database Load Balancer (e.g., HAProxy) — sits in front of replicas, routes reads automatically.
  • Weighted DNS Routing — DNS-level routing with weights per replica.
  • Multiple Connection Strings — app code has multiple connection strings; round-robins between them.
  • Managed Cloud DB (e.g., Amazon Aurora) — built-in reader endpoint that load-balances reads automatically.

Amazon Aurora

  • PostgreSQL/MySQL-compatible managed database on AWS.
  • Built-in reader endpoint that automatically routes reads across replicas.
  • Configurable as provisioned (fixed instance size) or serverless (pay per usage).
  • Handles most of the clustering complexity for you.

8. Multi-Master (Active-Active) Clustering

What It Is

  • Multiple write nodes — you can write to any node.
  • Each write is replicated to some (not all) other nodes.
  • Example: 3-node cluster → write to Node 1 → replicated to Node 2 (not Node 3).

Why Partial Replication Helps

  • With full 1:1 replication, every write costs double — no net gain.
  • With partial replication across 3 nodes: write to Node 1 → replicated to Node 2 (2 of 3 nodes process it).
  • Net effect: more total write throughput because not every write hits every node.
  • This is the underlying principle that makes sharding possible — selective distribution of writes.

Challenges of Multi-Master

  • Write conflicts — two nodes accept a write to the same record simultaneously → what's the correct value?
  • Ordering guarantees — transactions that depend on ordering become complex.
  • Consistency management — harder to guarantee all nodes agree.
  • Complex quorum settings — how many nodes must confirm before a write is considered committed.
Caleb's Advice Don't go multi-master until you absolutely need it (enterprise scale). Cache → read replicas → consider NoSQL before going multi-master on a relational DB. NoSQL databases (Cassandra, DynamoDB) are designed for this from the ground up.

9. Sharding

Sharding is splitting data across multiple nodes where each node holds a different subset of data — unlike full replication where every node has all the data.

Full Replication vs. Sharding

  • Full Replication: Node 1, Node 2, Node 3 → all hold [All Data]. Exact copies.
  • Sharding: Node 1 → [Users 1–1M]. Node 2 → [Users 1M–2M]. Node 3 → [Users 2M–3M].

Why Sharding Actually Scales Writes

  • Writes go to the appropriate shard based on the data — not all shards.
  • A write to Shard A doesn't replicate to Shards B and C (they don't have that data).
  • You can process genuinely more writes in parallel across shards.
  • This is how you actually increase write throughput — unlike 1:1 replication.
  • Each shard can still use replication internally — but it's only replicating a subset of the data.

Key Sharding Configurations

  • Shard key — what field determines which shard data goes to (e.g., user ID range).
  • Replication factor — how many copies of each shard exist for redundancy.
  • Quorum settings — how many nodes must confirm a read/write for it to count.
Coming Up Sharding is covered in much more depth in the upcoming NoSQL-focused videos. NoSQL databases (Cassandra, MongoDB) handle sharding far more gracefully than relational DBs.

10. Full DB Scaling Progression

  • Step 1 — Vertical scale the single instance. Tune CPU, RAM, disk before anything else.
  • Step 2 — Reduce load. Add Redis caching, offload specialized workloads to NoSQL, move analytics to OLAP.
  • Step 3 — Add a standby node. Removes SPOF, achieves High Availability and automatic failover.
  • Step 4 — Add read replicas. Scales reads horizontally, enables geographic distribution of reads.
  • Step 5 — Tune and add more read replicas. Scale each independently based on read load.
  • Step 6 — Multi-master clustering. Scales writes. Complex, enterprise-level. Consider NoSQL first.
  • Step 7 — Sharding. True horizontal partitioning of data. NoSQL-native or complex on relational DBs.

Key Takeaways

  • Vertical scale first — don't prematurely complicate with multi-node until the single node is well-tuned.
  • Replication ≠ write scaling — 1:1 full replication doubles writes per operation; net throughput stays the same.
  • Replication enables read scaling — read replicas let you scale reads horizontally.
  • Standby = HA, not performance — standby nodes are for failover, not added throughput.
  • Eventual consistency is the default for read replicas — synchronous replication is available but adds latency.
  • Sharding = actual write scaling — splitting data across nodes so writes don't replicate everywhere.
  • Multi-master is hard on relational DBs — NoSQL is better suited for it out of the box.
  • Don't over-engineer — cache → read replicas → consider NoSQL before going multi-master on relational.
