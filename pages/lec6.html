Lecture 6

CAP Theorem

Consistency, Availability & Partition Tolerance — the fundamental tradeoff in distributed databases

Overview

  • A fundamental principle describing the inherent tradeoffs in distributed database systems.
  • At first it seems complex, but the practical takeaway is simple: you can't have everything — you have to choose.
  • Only applies to distributed database systems — i.e., databases with multiple nodes.
  • On a single-node DB, these tradeoffs don't exist — you always have the latest data. The problem arises when multiple nodes need to stay in sync.

1. The Three Letters: C, A, P

C — Consistency

  • Every read returns the most up-to-date data.
  • If you write to Node 1, a subsequent read from Node 2 must return that same updated value.
  • All nodes serve identical, synchronized data at all times.
  • This is strong consistency — there is never a moment where two nodes have different data.

A — Availability

  • Every request gets a response — no errors, no timeouts.
  • The system always responds to requests.
  • It doesn't guarantee the data is the most recent — just that some data is returned.

P — Partition Tolerance

  • The system continues to function even when communication between nodes fails.
  • A network partition = a communication failure between nodes (broken link, dropped packets, network outage).
  • Nodes can't talk to each other → they fall out of sync. Partition tolerance means the system keeps running despite this.
  • This is not optional — network partitions are inevitable in any real distributed system.
Key Insight Since network partitions WILL happen, P is not a choice you make — it's a requirement. This is why the real decision always comes down to C vs. A.

2. The Core Conflict

Imagine a 3-node cluster and a partition happens — Node 3 can no longer communicate with Nodes 1 and 2. Node 3 has outdated data. A client reads from Node 3. What do you do?

  • Return outdated data — client gets a response, but it may be stale. Preserves Availability ✅, breaks Consistency ❌.
  • Return an error — client gets no data, but you never lie about state. Preserves Consistency ✅, breaks Availability ❌.
The Tradeoff You can't do both. If you want to always respond → you might serve stale data. If you want to always serve fresh data → you might have to go offline. This is the CAP tradeoff.

3. The Three Combinations

AP — Available + Partition Tolerant

Stays available during partition. May return stale data. Nodes eventually sync when partition heals.

Eventual Consistency

Cassandra, DynamoDB, CouchDB

CP — Consistent + Partition Tolerant

Goes offline during partition rather than serve stale data. Every read returns the most recent write.

Strong Consistency

HBase, Zookeeper, MongoDB

CA — Consistent + Available

No partition tolerance. Theoretical in distributed systems — only achievable on a single-node setup.

Not a real distributed choice

PostgreSQL / MySQL (single node)

AP — Available + Partition Tolerant (Detailed)

  • System stays available even during a partition — reads may return stale data temporarily.
  • Nodes eventually sync when the partition heals → this is called eventual consistency.
  • Best for: non-critical apps where stale data is acceptable — blogs, social media feeds, product catalogs, news feeds, shopping carts.

CP — Consistent + Partition Tolerant (Detailed)

  • System prioritizes correctness over availability.
  • During a partition, goes offline (unavailable) rather than serve stale data.
  • Best for: apps where data accuracy is critical — being wrong is worse than being down.
  • Banking, finance, military systems, medical records, inventory systems (can't sell the same item twice).

CA — Consistent + Available (Detailed)

  • Every read returns the most recent data AND every request gets a response — because there is never a partition.
  • Theoretical in distributed systems — in practice, you can't guarantee nodes will never lose communication.
  • Only really achievable on a single-node system (not distributed).
  • Worth knowing conceptually but not a real choice in distributed architecture.

4. Decision Framework

Ask yourself one key question:

The Question "If a network partition happens and nodes go out of sync, do I serve the request with potentially outdated data — or do I refuse to serve it and go offline?"
  • Serve it — stale response is fine → choose AP.
  • Refuse it — stale data is unacceptable → choose CP.

AP vs. CP Quick Reference

  • AP during partition: returns (possibly stale) data. Consistency type: Eventual. Always up. Use case: social media, blogs, feeds.
  • CP during partition: returns error / goes offline. Consistency type: Strong. May go down. Use case: finance, banking, military.
Mental Anchor Finance → CP. Social media → AP. Use this as your default starting point.

5. Advanced: Quorums & Partial Consistency

  • It doesn't always have to be all-or-nothing — a more nuanced approach uses quorums.
  • Instead of requiring ALL nodes to agree (strong) or just ONE node (eventual), require a majority (quorum) to agree.
  • Example: 3-node cluster → require 2/3 nodes to confirm a write before acknowledging it.
  • Gives you a tunable middle ground between full consistency and full availability.
  • Related concepts: database consensus algorithms — Paxos, Raft.
  • Covered in depth in future database clustering videos.

6. Real-World Database Examples

  • PostgreSQL (single node) — CA. No distribution → no partition concern.
  • MySQL (single node) — CA. Same as above.
  • Cassandra — AP. Tunable consistency; designed for availability.
  • DynamoDB — AP (default). Can be configured for stronger consistency.
  • HBase — CP. Consistency-first design.
  • Zookeeper — CP. Used for distributed coordination.
  • MongoDB — CP (default). Strong consistency with replica sets.
  • CouchDB — AP. Eventual consistency with conflict resolution.

7. CAP vs. ACID — Don't Confuse Them

  • CAP Theorem — applies to distributed database systems at the network/architecture level. "Consistency" means all nodes have the same data.
  • ACID — applies to individual database transactions at the query level. "Consistency" means the DB moves from one valid state to another.
  • These are separate concepts — same word, completely different meanings in each context.

Key Takeaways

  • P is not optional — network partitions always happen; you always need partition tolerance.
  • The real choice is C vs. A — do you prioritize correctness or uptime during a failure?
  • CP = error on partition — system goes down rather than lie about the data.
  • AP = stale data on partition — system stays up but may serve outdated results.
  • CA = single node — theoretical in distributed systems; not a real distributed choice.
  • No right or wrong answer — it completely depends on your application's requirements.
  • Finance → CP. Social media → AP. Use this as your mental anchor.
