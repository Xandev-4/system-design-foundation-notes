Lecture 2

Essential System Design Concepts

Distributed Systems, Availability, Consistency, Scaling & More

Overview

  • This video is a foundational overview of system design terminology.
  • You don't need to master every term deeply — the goal is that when these words show up in future lessons, you already have a mental model for them.
  • Think of it as a 1000-foot view of everything to come.

1. Distributed Computing

  • Running an application across multiple computers (nodes/servers) instead of just one.
  • Not limited to any single layer — applies to the backend, database, frontend, etc.
  • All computers must work together to produce a single result.
  • Introduces many new challenges — consistency, availability, fault tolerance, etc.
  • Most of the concepts in this video are built on top of distributed computing.
Example Instead of one server handling all requests, you have 3 servers — each handling part of the load and cooperating.

2. High Availability (HA)

  • The ability of a service to remain accessible as close to 24/7 as possible.
  • True 100% uptime is impossible — the goal is to get as close as possible.
  • If one node goes down, traffic is automatically rerouted to healthy nodes.
  • 4 nines (99.99%) is a good baseline benchmark — less than 1 hour downtime/year.
  • For critical systems (medical, financial, military), you'll need more nines.
  • More nines = exponentially more cost + implementation complexity — not just servers, but skilled devs to maintain them.

The Nines Table

  • 1 nine (90%) — ~36.5 days downtime/year
  • 2 nines (99%) — ~3.65 days downtime/year
  • 3 nines (99.9%) — ~8.76 hours downtime/year
  • 4 nines (99.99%) — ~52.6 minutes downtime/year ✓ target
  • 5 nines (99.999%) — ~5.26 minutes downtime/year

3. Reliability

  • A service's ability to actually do what it's supposed to do — not just whether it's up, but whether it works correctly.
  • A service can be highly available (99.999% uptime) but have 0% reliability if it's broken or losing data.
  • Reliability is harder to measure than availability — it's not just nines, it's about defining what correct behavior looks like for your specific app.
Analogy A friend who always picks up your calls (available) but never actually shows up when you need help moving (not reliable). Two different things.

Availability vs. Reliability

  • Availability — can you access the service?
  • Reliability — does the service work correctly?

4. Consistency

  • Every client sees the same data at the same time across all nodes.
  • Only becomes a concern in distributed systems — on a single server, you're always 100% consistent by default.
  • The problem arises when data written to one node needs to propagate to other nodes — there's a window where data can be out of sync.

Strong Consistency

  • A write is held in a pending state until all nodes have applied it — only then is it confirmed.
  • Benefit: all nodes always have the latest data — no stale reads.
  • Cost: slower writes — you must wait for propagation to all nodes.

Eventual Consistency

  • A write is confirmed immediately on the write node, then propagates to other nodes over time.
  • Benefit: much faster writes, lower latency.
  • Cost: brief window where different clients can see different (stale) data.

When to Use Which

  • Strong consistency → banking, financial transactions, military, medical real-time data. Stale data could allow double-spending or catastrophic decisions.
  • Eventual consistency → social media, basic CRUD apps, content feeds. A post not appearing for 2 seconds is not a big deal.
Heads Up Strong consistency can also harm availability — this ties directly into the CAP Theorem, covered in a future video.

5. Scalability

  • The ability of an app to support an increase in workload — traffic spikes, user growth, seasonal peaks like Black Friday — without breaking.
  • Simplest approach: over-provisioning — having more hardware than you typically need so you have headroom.
  • You need to plan for both typical usage and peak usage (e.g., 8 AM spikes every day).

Vertical Scaling

Add more resources to the existing server — more CPU, RAM, faster storage.

aka "scale up"

Has a hardware ceiling

Horizontal Scaling

Add more servers running in parallel. Can combine with vertical scaling.

aka "scale out"

No hard ceiling, preferred at scale

Elasticity

Automatically scale up AND down based on current load. Handled by cloud providers.

AWS Auto Scaling, etc.

Always automatic

Key Distinction Scalability ≠ High Availability. A super powerful single server can be highly scalable via vertical scaling and still be a Single Point of Failure if it goes down.

6. Elasticity

  • The ability of a system to automatically scale up AND down based on current workload.
  • Scalability is typically about handling increases in load (sometimes manually) — elasticity does both directions, always automatically.
  • Usually provided by cloud providers (AWS, GCP, Azure) — they handle resource allocation behind the scenes via virtualization.
  • No technicians manually adding/removing hardware — the cloud provider pools servers and reallocates virtual resources dynamically.

Scalability vs. Elasticity

  • Scalability — handles increased workload; can be manual or automatic; typically one direction (up).
  • Elasticity — dynamically matches current load; always automatic; scales up AND down.

7. Single Point of Failure (SPOF)

  • Any component in the system that, if it fails, brings down the entire service because there's no backup.
  • Like the weakest link in a chain — the chain is only as strong as that one link.
  • Example: frontend and backend have redundancy but the database does not → the database is the SPOF.
  • Identifying and eliminating SPOFs is a core task when building highly available systems.

8. Redundancy

  • Having backup/duplicate components ready to take over if the primary fails.
  • In everyday life "redundant" = unnecessary. In computing, redundant = good.
  • A standby server doesn't handle normal traffic — it only activates if the primary fails.
  • Adds cost (maintaining duplicate systems) but greatly improves availability.
  • Redundancy is the primary tool for eliminating Single Points of Failure.
Analogy A spare tire — you don't use it every day, but it saves you when you need it.

9. Fault Tolerance

  • A system's ability to continue functioning even when something goes wrong.
  • Achieved by removing SPOFs — usually via redundancy.
  • If your primary DB fails and the standby is promoted automatically → the system is fault tolerant.
  • Fault Tolerance = the result of designing with redundancy and eliminating SPOFs.
The Chain Remove SPOF → Add Redundancy → Achieve Fault Tolerance

10. Replication

  • The process of creating copies (replicas) of a node/database so data exists in multiple places.
  • A node = one server/computer in a multi-server setup (e.g., one database instance).
  • Replication creates a duplicate node from the primary.
  • The standby database in the redundancy example is a replica of the primary.

11. Database Clusters

A group of multiple database nodes working together.

Common Setup: Primary/Replica Architecture

  • 1 Write (Master/Primary) Node — handles all write operations.
  • Multiple Read (Replica) Nodes — handle read operations only.
  • You can have a standby write node for redundancy — idle until the primary fails.

Why This Architecture?

  • Most apps are read-heavy (~80% reads, ~20% writes).
  • You can scale read nodes independently to handle more read load.
  • Single write node = no write conflict issues (keeps write logic simple).

Multi-Master Setup

  • Multiple write nodes = more complexity — write conflicts, ordering issues.
  • Covered in more advanced database clustering topics in later videos.

12 & 13. Replication Types

Synchronous Replication

  • A write is not confirmed until ALL replica nodes have written the data.
  • Equivalent to strong consistency.
  • Guarantees no stale reads.
  • Downside: higher latency — writes take longer.

Asynchronous Replication

  • The write is confirmed immediately on the primary — replicas receive the data later.
  • Equivalent to eventual consistency.
  • Benefit: low latency writes.
  • Downside: briefly inconsistent — replicas may serve stale data.
Practical Tip Most apps are read-heavy. Use multiple read replicas + single write node. Add a standby write node for redundancy. Use async replication unless your use case requires strong consistency.

14. Latency

  • The time from a request to a response — how long something takes.
  • Can be measured at any level: network latency, DB query latency, API response latency.
  • Strong/synchronous consistency increases latency — writes must wait for propagation.
  • High latency = worse user experience.
Example You request a webpage → it loads in 100ms → that 100ms is the latency.

15. ACID Properties

A set of four properties that relational databases enforce to keep data correct and trustworthy. Most SQL databases (PostgreSQL, MySQL) are ACID compliant by default.

Watch Out ACID "consistency" ≠ distributed systems "consistency". Same word, completely different meanings. Don't mix them up in interviews.

A — Atomicity

  • A transaction is treated as a single unit — it either fully succeeds or fully fails.
  • No partial completions — if part of a transaction fails, the whole thing is rolled back.
  • Example: Transfer money → deduct from Account A AND credit Account B → if credit fails, deduction is also reversed.

C — Consistency (ACID)

  • The database moves from one valid state to another valid state.
  • Every change must respect all integrity rules defined in the database.
  • Example: Referential integrity — you can't insert a record that references a foreign key that doesn't exist.

I — Isolation

  • Concurrent transactions are isolated from each other — execute independently as if running sequentially.
  • One transaction's intermediate state is not visible to other transactions.
  • Prevents race conditions and dirty reads.

D — Durability

  • Once data is committed, it is permanently saved — even if the power goes out immediately after.
  • Committed state = a checkpoint that survives crashes, restarts, and failures.

16. Sharding

  • Splitting a single database into multiple smaller pieces (shards) distributed across multiple servers.
  • Used when one database gets too large or handles too much traffic for a single instance.
  • Each shard holds a subset of the data.
  • Splits into two types: Vertical Partitioning and Horizontal Partitioning.

17. Vertical Partitioning

  • Splitting a database by columns — i.e., by feature or domain.
  • Group related columns into separate tables or even separate databases.
  • Example: User profile data → one database; user payment data → another database.
  • Good for splitting a monolithic DB into microservice-style databases.
Structure [User DB] id, name, email — [Orders DB] id, user_id, item — [Payments DB] id, user_id, amount

18. Horizontal Partitioning (Sharding)

  • Splitting a database by rows — the same table structure exists across all shards, but each holds a different subset of rows.
  • Example: Users with IDs 1–1M → Shard A; IDs 1M–2M → Shard B; IDs 2M–3M → Shard C.
  • Good for handling massive datasets where one table becomes too large for a single server.
  • Adds complexity — queries across shards are harder.

Vertical vs. Horizontal Partitioning

  • Vertical — split by columns/features; different schema per partition; separates features/microservices.
  • Horizontal — split by rows/data ranges; same schema, different data; handles massive row counts.

Key Tradeoffs to Remember

  • Strong Consistency ↔ Higher Latency — waiting for all nodes means slower writes.
  • Eventual Consistency ↔ Stale Data Risk — faster writes but temporary inconsistency.
  • More Nines of Availability ↔ Exponentially More Cost & Complexity.
  • Redundancy ↔ Increased Infrastructure Cost.
  • Strong Consistency ↔ Potentially Lower Availability — CAP Theorem, covered in future videos.

Quick Recap — All Concepts

  • Distributed Computing — running a service across multiple machines.
  • High Availability — maximizing uptime, measured in nines. Target: 4 nines.
  • Reliability — service does what it's supposed to do.
  • Consistency — all nodes serve the same data at the same time.
  • Strong Consistency — confirmed only after all nodes are updated.
  • Eventual Consistency — confirmed immediately; nodes sync later.
  • Scalability — ability to handle increased workload.
  • Vertical Scaling — upgrade existing server (scale up).
  • Horizontal Scaling — add more servers (scale out).
  • Elasticity — auto-scale up AND down based on load.
  • SPOF — any component with no backup that can take down the whole system.
  • Redundancy — backup components ready to replace failed ones.
  • Fault Tolerance — system continues working despite failures.
  • Replication — copying data/nodes to create redundancy.
  • Database Cluster — group of database nodes working together.
  • Synchronous Replication — write confirmed only after all replicas acknowledge.
  • Asynchronous Replication — write confirmed immediately; replicas sync later.
  • Latency — time from request to response.
  • ACID — Atomicity, Consistency, Isolation, Durability.
  • Sharding — splitting a DB across multiple servers for scale.
  • Vertical Partitioning — split DB by columns/features.
  • Horizontal Partitioning — split DB by rows/data ranges.
