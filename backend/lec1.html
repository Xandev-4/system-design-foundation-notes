Topic 01

Backend from First Principles

Roadmap · Series by Sriniously · Last updated: May 2026

Overview

Backend engineering is far more than building CRUD APIs. It is about building reliable, scalable, fault-tolerant and maintainable codebases and efficient systems. This series covers backend from foundational first principles — language and framework agnostic.

Goal Understand the why behind every backend concept so knowledge transfers across any language or framework.

A high-level understanding starts with how a browser request flows through the network, firewalls, DNS, and arrives at a remote server — and how the server forms and sends back a response.

● Presentation Layer

Deals with accepting and sending user data.

Routing, Middlewares, Handlers, Validation

● Business Logic Layer

Core logic of your application.

Services, Domain Models, BLL Rules

● Data Access Layer

Communicates directly with the database.

ORMs, Raw Queries, Migrations

HTTP Protocol

HTTP defines how clients and servers communicate. Understanding it deeply is a prerequisite for all other backend concepts.

Core Concepts

  • HTTP Methods — GET, POST, PUT, DELETE and their semantics and principles
  • CORS — How it works, simple requests vs preflight requests, request and response flow
  • HTTP Responses — Response structure, status codes (2xx, 4xx, 5xx) and when to use each
  • HTTP Caching — ETags, max-age headers, different caching techniques
  • HTTP Versions — Differences between HTTP/1.1, HTTP/2.0 and HTTP/3.0

Advanced HTTP

  • Content Negotiation — Client-server negotiation using headers
  • Persistent Connections — How keep-alive connections work
  • Compression — gzip, deflate, Brotli and trade-offs
  • SSL / TLS / HTTPS — Security layer on top of HTTP
  • HTTP Headers — Request headers, representational headers, general headers, security headers

Routing

Routing maps URLs to server-side logic. It is the entry point of every request into your application.

Route Types

  • Static Routes — Fixed URL paths
  • Dynamic Routes — URL with path parameters e.g. /users/:id
  • Nested Routes — Hierarchical route structures
  • Catchall / Wildcard Routes — Match anything not caught by specific routes
  • Regex-based Routes — Pattern matching on URL segments

Route Components & Best Practices

  • Path Parameters & Query Parameters — Differences and use cases
  • API Versioning — URI versioning, header versioning, query string, media type versioning
  • Deprecating Routes — Best practices for retiring old API versions
  • Route Grouping — Benefits for versioning, permissions and shared middleware
  • Securing Routes — Auth-gated routes and performance optimisation of route matching

Serialisation & Deserialisation

Serialisation translates native data into a transmittable format. Deserialisation is the reverse — converting incoming data back into native types.

Formats

  • Text-based — JSON, XML; human readable, easier to debug but slower
  • Binary — Protobuf; faster and smaller payload, but not human readable

JSON Deep Dive

  • Data Types — Strings, numbers, booleans, arrays, objects
  • Nested Objects & Collections — How they are handled during (de)serialisation
  • Common Errors — Missing/extra fields, null values, date serialisation, timezone issues
  • Custom Serialisation — Implementing custom logic before serialising
  • Security Concerns — Injection attacks, validate JSON schemas before processing
  • Performance — Reduce payload via compression, eliminate unnecessary fields
Trade-off Text formats (JSON) are readable and easy to debug. Binary formats (Protobuf) are faster but opaque — choose based on your performance requirements.

Authentication & Authorisation

Authentication answers who are you? Authorisation answers what are you allowed to do?

Authentication Methods

  • Stateful — Sessions, server stores session state
  • Stateless — JWT (JSON Web Tokens), token is self-contained
  • Basic Auth — Username/password in headers (base64 encoded)
  • Bearer Token — Token passed in Authorization header
  • API Keys — Simple key-based authentication for services
  • OAuth 2.0 — Delegated access protocol, deep dive
  • OpenID Connect — Identity layer on top of OAuth 2.0
  • Multi-Factor Authentication — How MFA works

Security Concepts

  • Salting & Hashing — Cryptographic techniques for password storage
  • RBAC / ABAC / ReBAC — Role-based, attribute-based, relationship-based access control
  • Securing Cookies — HttpOnly, SameSite, Secure flags
  • CSRF / XSS / MITM — Common attacks and how to prevent them
  • Audit Logging — Recording auth events for monitoring
  • Timing Attacks — Exploiting response time differences to infer valid credentials
  • Rate Limiting & Account Lockout — Defending against brute-force

Validation & Transformation

Validation is the gateway to your business logic. Server-side validation is mandatory even when client-side validation exists.

Types of Validation

  • Syntactic Validation — Is this a valid email, phone number, date format?
  • Semantic Validation — Date of birth cannot be in the future; age between 1–120
  • Type Validation — Is this a string, integer, array, object?
  • Relationship Validation — Password must match confirm password
  • Conditional Validation — Partner name required only if married = true
  • Chain Validation — Lower case → remove special chars → check length

Transformations

  • Type Casting — Convert query param strings to expected types
  • Date Format Normalisation — Align to expected timestamp format
  • Normalisation — Lowercase emails, trim whitespace, add country code to phone numbers
  • Sanitisation — Prevent SQL injection and XSS via input cleaning

Best Practices

  • Fail Fast — Return early, reduce unnecessary processing
  • Aggregate Errors — Return all validation errors in a single response
  • Obfuscate Messages — Say invalid credentials, not invalid password
  • Consistency — Keep frontend and backend validation in sync

Middlewares

Middleware sits between the incoming request and the final handler. It is chained in sequence and each piece passes control to the next via a next() function.

Common Middleware Types

  • Security Middleware — Adds headers: X-Content-Type, Strict-Transport-Security, CSP
  • CORS Middleware — Applies CORS headers to every request/response
  • CSRF Middleware — Prevents cross-site request forgery
  • Rate Limiting Middleware — Throttles request frequency per client
  • Authentication Middleware — Reusable route-protecting logic
  • Logging Middleware — Request logging, structured logging for observability
  • Error Handling Middleware — Catches and formats application-level errors
  • Compression Middleware — Compresses response bodies to reduce bandwidth
  • Body Parsing Middleware — Parses JSON, URL-encoded forms, multipart file uploads

Ordering & Best Practices

  • Correct order: Log → Authenticate → Validate → Handle Route → Error Handle
  • Exit Early — Middleware can short-circuit the pipeline
  • Lightweight — Keep middlewares efficient; order affects both performance and security

Request Context

Request context is metadata passed through middlewares, controllers and services — a temporary, request-scoped state valid only for the duration of that single request.

Components of Request Context

  • Request Metadata — HTTP method, URL, headers, query params, body
  • Session & User Info — User data injected by auth middleware for that request scope
  • Tracking & Logging — Unique request IDs, Trace IDs for distributed tracing
  • Request-Specific Data — Custom data injected during request lifecycle

Timeouts & Best Practices

  • Timeout Types — Request timeouts, custom timeouts, cancellation signals
  • Lightweight — Prevent memory overhead by keeping context lean
  • Cleanup — Context data must be cleared after request lifecycle to prevent memory leaks
  • Avoid Over-reliance — Don't tightly couple components through context

Handlers, Controllers & Services

The MVC pattern separates concerns into Models, Views and Controllers. Each layer has a distinct responsibility.

Responsibilities

  • Handler / Controller — Accepts request, calls service, returns formatted response
  • Service — Contains business logic; called by controller
  • Model — Represents core domain entities (User, Order, Product)

Best Practices

  • Reduce Code with Middleware — Centralise repeated logic into middleware
  • Centralised Error Handling — Handle errors in one place per layer
  • Consistent Response Formats — Standardise success and error message shapes

CRUD Deep Dive

CRUD operations map directly to HTTP methods. Understanding the semantics and best practices for each is essential for building clean APIs.

● Create / POST

For resource creation and submissions.

201 Created · 400 Bad Request

● Read / GET

Fetch a list or a single resource.

200 OK · 404 Not Found

● Update / PUT & PATCH

PUT replaces, PATCH partially updates.

200 OK · 204 No Content

Advanced CRUD Patterns

  • Pagination — Offset-based and cursor-based pagination strategies
  • Search API — Implementing full-text and filtered search
  • Sorting & Filtering — Query param patterns for ordering and narrowing results
  • Best Practices — Strict validation, consistent formatting, limiting payload, redacting sensitive fields

RESTful Architecture

REST is a set of architectural constraints for designing APIs around resources using HTTP semantics consistently.

Core Principles

  • Resource-centric Design — APIs designed around nouns (resources), not verbs
  • HTTP Semantics — Stick to correct method usage
  • Filtering & Pagination — Standardised query param conventions
  • Versioning Strategies — URI, header, query string, media type versioning
  • Content Negotiation — Serve different formats based on Accept headers
  • OpenAPI Spec — Design APIs with documentation in mind from the start
  • Client-side Caching — ETags and conditional requests for caching support
  • Large Request/Response Optimisation — Pagination, streaming, compression

Databases

Choosing the right database and understanding how to use it efficiently is one of the most critical skills in backend development.

Types & Theory

  • Relational vs Non-relational — SQL vs NoSQL: differences and when to use each
  • ACID — Atomicity, Consistency, Isolation, Durability
  • CAP Theorem — Consistency, Availability, Partition tolerance trade-offs

Design & Optimisation

  • Schema Design — Normalisation, relationships, constraints
  • Indexing — Speeding up reads on frequently queried fields
  • Query Optimisation — Avoiding N+1 problems, proper use of joins, lazy loading
  • Caching at DB Level — Connection pooling, query result caching
  • Data Integrity — Constraints, validations, transactions, concurrency
  • ORMs — When to use an ORM and what the trade-offs are
  • Migrations — Managing schema changes safely over time

Business Logic Layer (BLL)

The BLL is the middle layer in a three-tier architecture. It holds your application's core rules and sits between the presentation layer and the data access layer.

Design Principles

  • Separation of Concerns — Each layer has one job
  • Single Responsibility — Each class/module does one thing
  • Open/Closed Principle — Open for extension, closed for modification
  • Dependency Inversion — Depend on abstractions, not concrete implementations

BLL Components

  • Services — Encapsulate use-case logic
  • Domain Models — Core entities like User, Order, Product
  • Business Rules — Application-specific constraints and logic
  • Business Validation Logic — Domain-level checks beyond input validation
  • Error Propagation — How errors bubble up from service to presentation layer

Caching

Caching stores frequently accessed data closer to the consumer to reduce latency and database load.

Caching Types

  • In-Memory Caching — Fastest, process-level
  • Browser Caching — Client-side, controlled via HTTP headers
  • Database Caching — Query result caching in Redis or similar
  • Client-side vs Server-side — Where caching happens and the trade-offs

Strategies & Eviction

  • Caching Strategies — Cache-aside, Write-through, Write-behind, Read-through
  • Eviction Strategies — LRU, LFU, TTL, FIFO
  • Cache Invalidation — Manual, TTL-based, event-based invalidation
  • L1 vs L2 Cache — L1 = in-memory (fast, small); L2 = distributed (slower, large)

Transactional Emails

Transactional emails are programmatically triggered messages sent in response to user actions.

Anatomy of a Transactional Email

  • Subject — Clear and action-oriented
  • Preheader — Preview text visible in email clients
  • Body Header — Branding and context
  • Main Content — The actual message/information
  • CTA — The primary action button (Verify Email, Reset Password)
  • Footer — Legal, unsubscribe, contact info
  • Dynamic Parameters — Personalisation using template variables

Task Queuing & Scheduling

Task queues offload work from the request cycle — return a response immediately and process the task in the background.

Use Cases

  • Queuing — Sending emails, image processing, third-party API calls
  • Heavy Computation — Batch processing across multiple tables
  • Scheduling — DB backups, recurring notifications, data sync, log/cache cleanup

Task Queue Architecture

  • Components — Producer, Queue, Consumer, Broker, Backend
  • Dependencies — Chain dependency, parent-child relationships between tasks
  • Task Groups — Execute multiple tasks concurrently, wait for all to complete
  • Error Handling & Retries — Retry logic with backoff strategies
  • Task Prioritisation — Payment tasks before notification tasks
  • Rate Limiting — Control task throughput to avoid overwhelming downstream services

Elasticsearch

Elasticsearch is a distributed search and analytics engine built on Apache Lucene. Used when databases are too slow for full-text search.

How It Works

  • Inverted Index — Maps terms to the documents that contain them
  • TF-IDF — Term Frequency / Inverse Document Frequency relevance scoring
  • Segments & Shards — How data is stored and distributed

Use Cases & Searching

  • Use Cases — Typeahead, log analytics, social media full-text search
  • Full-Text Search — Analysed text with relevance scoring
  • Advanced Patterns — Filtering, aggregation, fuzzy search
  • Optimisation — text vs keyword fields, analyzers, boosting, pagination

Error Handling

Proper error handling is what separates production-grade systems from hobby projects.

Error Types

  • Syntax Errors — Code that cannot be parsed
  • Runtime Errors — Failures during execution (null reference, network timeout)
  • Logical Errors — Code runs but produces wrong results

Strategies & Practices

  • Fail Safe vs Fail Fast — Fail fast reduces blast radius; fail safe tries to recover
  • Graceful Degradation — System continues with reduced functionality rather than crashing
  • Don't Swallow Errors — Always log or propagate; never silently catch and ignore
  • Custom Error Types — Domain-specific error classes for better handling
  • Global Error Handler — Centralised handler for unhandled rejections
  • User-Facing Errors — Friendly messages; no stack traces to users
  • Monitoring & Alerting — Sentry, ELK stack; email/Slack alerts

Config Management

Config management decouples environment-specific settings from application logic.

Config Types

  • Static Configs — DB credentials, API endpoints (change rarely)
  • Dynamic Configs — Feature flags, rate limits (change at runtime)
  • Sensitive Configs — Credentials, tokens, secrets (never commit to VCS)

Sources & Best Practices

  • Sources — .env files, JSON, YAML config files
  • Env Vars vs CLI Flags vs Static Files — Trade-offs between each approach
  • Best Practices — Never hardcode secrets, use secret managers in production, validate config on startup

Logging, Monitoring & Observability

You cannot fix what you cannot see. Observability is the ability to understand internal system state from external outputs.

Logging

  • Log Types — System logs, application logs, access logs, security logs
  • Log Levels — DEBUG, INFO, WARN, ERROR, FATAL and when to use each
  • Structured vs Unstructured — Structured (JSON) is machine-parseable; unstructured is human-readable
  • Best Practices — Centralised logging, log rotation, contextual logs, never log passwords or API keys

Monitoring & Observability

  • Types — Infrastructure monitoring, APM, uptime monitoring
  • Tools — Prometheus, Grafana
  • Three Pillars — Logs, Metrics, Traces
  • Best Practices — Correlate logs with trace IDs, ensure security and compliance of log data

Graceful Shutdown

Graceful shutdown ensures a server finishes in-flight requests before terminating — rather than cutting connections abruptly.

Shutdown Steps

  • 1. Capture Signal — SIGTERM, SIGINT or SIGKILL from OS
  • 2. Stop Accepting Requests — Close the listening socket
  • 3. Complete In-flight Requests — Wait for active requests to finish
  • 4. Close External Resources — DB connections, open files, message queue connections
  • 5. Terminate — Exit the process cleanly

Security

Security is not a feature — it is a foundational concern that must be built in from the start.

Common Attack Vectors

  • SQL Injection — Malicious SQL via unsanitised input
  • XSS — Injecting scripts into web pages viewed by other users
  • CSRF — Tricking users into performing unintended actions
  • Broken Authentication — Weak tokens, poor session management
  • Insecure Deserialisation — Exploiting deserialisation of untrusted data

Secure Design Principles

  • Least Privilege — Grant minimum permissions necessary
  • Defence in Depth — Multiple independent security layers
  • Fail Secure — Default to denial on failure
  • Security by Design — Build security in, not on top
  • Input Validation & Sanitisation — Treat all input as untrusted

Scaling & Performance

Performance is measured before it is optimised. Profile first, then fix.

Performance Metrics

  • Response Time — P50, P95, P99 latencies
  • Resource Utilisation — CPU, memory, disk, network
  • Bottleneck Identification — Profiling to find the slowest parts of the stack

Optimisation Techniques

  • Caching — Reduce redundant computation and DB hits
  • DB Optimisation — Avoid N+1 queries, use proper joins
  • Indexing — Index foreign keys and frequently searched fields
  • Batch Processing — Minimise DB round-trips with bulk operations
  • Offload Non-critical Tasks — Push emails, logging to background queues

Concurrency & Parallelism

These are related but distinct concepts that are frequently confused.

● Concurrency

Dealing with multiple tasks at once by interleaving execution.

Best for I/O-bound tasks (network, disk)

● Parallelism

Actually executing multiple tasks simultaneously on multiple cores.

Best for CPU-bound tasks (computation)

● Node.js Model

Single-threaded event loop — concurrent but not parallel by default.

Worker threads for CPU-bound work

Object Storage & Large Files

Object storage is used for unstructured data like images, videos, backups and documents.

  • Use Cases — User uploads, media assets, backups, static file hosting
  • AWS S3 — The most widely used object storage service
  • Large File Handling — Chunking and streaming to avoid loading entire files into memory
  • Multipart File Uploads — Upload large files in parts for reliability and resumability

Real-Time Backend Systems

Real-time systems push data to clients as it changes, rather than requiring the client to poll for updates.

  • WebSockets — Full-duplex persistent connection between client and server
  • Server-Sent Events (SSE) — Unidirectional push from server to client over HTTP
  • Pub/Sub Architecture — Publish-subscribe pattern for decoupled real-time event distribution

Testing & Code Quality

Untested code is code that will break in production at the worst possible time.

Test Types

  • Unit Testing — Test individual functions/modules in isolation
  • Integration Testing — Test how modules work together
  • End-to-End Testing — Test the full user journey through the system
  • Performance Testing — Measure throughput and latency under load
  • Load & Stress Testing — Find breaking points under extreme load
  • Security Testing — Identify vulnerabilities before attackers do

Code Quality

  • TDD — Write tests before code
  • CI/CD Test Automation — Run tests automatically on every push
  • Linting & Formatting — ESLint, Prettier, language-specific tools
  • Cyclomatic Complexity — Measures number of paths through a function; keep it low

12 Factor App

The 12 Factor App is a methodology for building modern, scalable, maintainable software-as-a-service applications.

Source Originally written by Heroku engineers. Widely adopted as a best-practice guide for cloud-native application design.
  • I. Codebase — One codebase, many deploys
  • II. Dependencies — Explicitly declare and isolate dependencies
  • III. Config — Store config in the environment
  • IV. Backing Services — Treat backing services as attached resources
  • V. Build, Release, Run — Strictly separate build and run stages
  • VI. Processes — Execute app as stateless processes
  • VII. Port Binding — Export services via port binding
  • VIII. Concurrency — Scale out via process model
  • IX. Disposability — Fast startup and graceful shutdown
  • X. Dev/Prod Parity — Keep development, staging and production similar
  • XI. Logs — Treat logs as event streams
  • XII. Admin Processes — Run admin/management tasks as one-off processes

OpenAPI Standards

OpenAPI is a specification for describing REST APIs in a machine-readable format. It powers documentation, code generation and testing tooling.

Key Concepts

  • Swagger → OpenAPI — Swagger was donated to the Linux Foundation and became OpenAPI Spec
  • Active Versions — OpenAPI 3.0 and 3.1 are currently in use
  • Document Structure — Metadata, Paths, Components, Security Definitions, Responses
  • Schemas — Data type definitions (reusable via $ref)

API First Development

  • Write the OpenAPI spec first before writing any code
  • Enables parallel frontend/backend development from a shared contract
  • Best practices: avoid duplication with $ref, document all error responses

Webhooks

Webhooks are server-initiated HTTP callbacks. Instead of polling, the external service pushes data to you when an event occurs.

Key Components

  • Webhook URL — Your publicly accessible endpoint that receives events
  • Event Triggers — The events that cause the webhook to fire
  • Payload — The data sent in the webhook request body
  • Response Handling — Return 200 quickly; process asynchronously

Best Practices

  • Signature Verification — Verify webhook origin using HMAC signature
  • HTTPS Only — Never accept webhooks over plain HTTP
  • Quick Response — Respond 200 immediately, push work to a queue
  • Retry Logic — Handle duplicate deliveries idempotently
  • Real-World Examples — Stripe payments, GitHub events, Slack, Discord, Twilio

DevOps for Backend Engineers

Backend engineers don't need to be DevOps specialists, but understanding the deployment pipeline is essential for shipping production-grade systems.

Core Concepts

  • CI — Automatically build and test on every commit
  • CD (Delivery) — Always have a deployable artifact ready
  • CD (Deployment) — Automatically deploy every passing build to production

Tools & Scaling

  • Docker — Containerise applications for consistent environments
  • Kubernetes — Orchestrate containers at scale
  • Infrastructure as Code — Define infrastructure in version-controlled config files
  • Horizontal Scaling — Add more instances (scale out)
  • Vertical Scaling — Add more power to existing instance (scale up)
  • Blue-Green Deployment — Switch traffic to new version atomically
  • Rolling Deployment — Gradually replace old instances — zero downtime
