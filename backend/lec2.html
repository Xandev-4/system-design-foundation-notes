What is a Backend, How Do They Work & Why Do We Need Them?

A holistic, ground-up view of backend systems — from DNS hops to browser sandboxes.

What is a Backend?

A backend, in its traditional definition, is a computer listening for requests — HTTP, WebSocket, gRPC, or any other protocol — through an open port accessible over the internet, so that clients or frontends can connect to it, send data, or receive data.

We call it a server because it serves content. That content can be static files like HTML, CSS, JavaScript, or images — or dynamic data like JSON. It also accepts incoming data from clients.

Core Idea A backend is a centralized computer that holds state and data for all users — serving and persisting information on their behalf.

How a Request Travels

When you type a URL into a browser, your request doesn't magically appear at the server. It takes a specific journey through several hops before it reaches your backend code.

● Step-by-Step Flow

  • Browser — initiates the request with a domain name
  • DNS Server — resolves the domain to an IP address via A records or CNAME records
  • Firewall (e.g. AWS Security Group) — checks if the request's port is allowed; blocks if not. Port 443 for HTTPS, 80 for HTTP must be explicitly permitted
  • EC2 / Server Instance — the physical or virtual machine your backend lives on
  • Reverse Proxy (e.g. Nginx) — sits in front of your server, manages redirects and SSL certificates, and forwards traffic to your app's local port (e.g. localhost:3001)
  • Your App Server — the final hop; Node.js, Go, or whatever is actually running your backend logic
Local Dev When developing locally, you skip DNS, firewall, and reverse proxy entirely — you just go straight to localhost:3000. Same server logic, fewer hops.

● DNS in Brief

  • A Record — maps a domain/subdomain directly to an IP address
  • CNAME Record — maps a subdomain to another domain name

● Reverse Proxy (Nginx)

  • Sits in front of one or more backend servers
  • Handles SSL termination — manages HTTPS certificates (e.g. via Certbot)
  • Redirects port 80 → 443 automatically
  • Routes incoming requests to the correct local port based on server_name config
  • Centralises config — you change one place instead of every individual server

Why Do We Need Backends?

Strip everything down to its core and the purpose of a backend is one word: data. The need to fetch data, receive data, and persist data — and every action that deals with that data.

● The Instagram Like Example

  • You tap like on a post
  • The app sends a request to the backend server
  • The server identifies who you are (your user ID)
  • It persists the like action in a database
  • It identifies the post owner and triggers a notification
  • The post owner receives a notification on their device

All of this requires a centralized computer that knows the state of all users simultaneously. Your phone only knows about you — the server knows about everyone.

Key Insight A backend is necessary any time you need to share state between multiple users or devices. Your frontend knows nothing about other users — the server knows everything.

How Frontends Work

A frontend server — like a Next.js app — follows the same DNS → firewall → Nginx journey. But what it serves is fundamentally different. Instead of processing logic and returning data, it returns files: HTML, CSS, JavaScript, fonts, images.

● Browser Rendering Flow

  • Browser fetches the primary HTML file
  • HTML references CSS, JS, fonts and images — each fetched in separate requests
  • CSS is parsed → browser paints the visual layout
  • JavaScript is fetched → browser hydrates event listeners and interactivity
  • Only now does the page become fully functional
Key Difference Backend: server runs the code, sends you the result. Frontend: server sends you the code, your browser runs it.

Why Not Write Backend Logic in the Frontend?

Since the browser is also a runtime environment, could we just connect directly to databases and run all logic client-side? Short answer: no. Here's why.

🔒 Security

Browsers are sandboxed — isolated from the OS, file system, and environment variables. Backend logic often needs to read log files, access secrets, and connect to internal services. Browsers are deliberately prevented from doing this.

e.g. You can't read /etc/secrets from a browser tab

🌐 CORS Restrictions

Browsers enforce CORS policy — JavaScript can only call APIs on the same domain unless the external API explicitly allows it via headers. Backend servers have no such restriction and can freely connect to any external service.

e.g. Calling a payment API or sending an email from frontend = blocked

🗄️ Database Access

Native database drivers (pg, mongodb, mysql2) are built for environments that can manage socket connections, binary data, and connection pools. Browsers cannot maintain persistent DB connections.

e.g. If every browser opened its own DB connection, the DB would be overwhelmed instantly

● Connection Pooling

Backend servers maintain a pool of reusable database connections rather than creating and destroying a connection per request. This is critical because backends receive thousands of requests per second — establishing a new DB connection for each one would overwhelm and crash the database server.

● Computing Power

  • Client devices vary wildly — from a flagship laptop to a 256 MB RAM smartphone
  • Running heavy business logic on the client means inconsistent performance for users
  • A centralized backend server can be scaled up (more CPU/RAM) or out (more instances) independently of client hardware
Browser Sandbox Browsers intentionally restrict what code can do. A browser fetches code from a remote server and executes it — if that sandbox didn't exist, any website you visit could read your entire file system and exfiltrate your data.

Mental Model

Think of backend systems in layers. Each layer has exactly one job, and understanding that separation is the foundation of everything that follows in this course.

  • Presentation Layer — accepts requests, sends responses (your routes and controllers)
  • Business Logic Layer — the core rules of your application (your services)
  • Data Access Layer — communicates with the database (your models/repositories)
