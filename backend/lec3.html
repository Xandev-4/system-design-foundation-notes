Understanding HTTP for Backend Engineers

A deep dive into HTTP — the medium through which browsers talk to servers.

HTTP Fundamentals

HTTP is the protocol through which browsers communicate with servers — either to send data or receive it. Two core ideas are at the heart of every HTTP interaction.

● Statelessness

HTTP has no memory of past interactions. Each request is self-contained and carries all the information the server needs — headers, URL, method, body. After responding, the server forgets the request entirely.

  • Simplicity — server does not need to store session state, reducing complexity
  • Scalability — requests can be distributed across any server since none of them track session state
  • Fault Tolerance — if a server crashes, no client state is lost because there was none
Practical Note Because HTTP is stateless, developers layer state management on top — cookies, sessions, and tokens — to maintain continuity for things like user logins and shopping carts.

● Client–Server Model

  • Client — browser or app that always initiates the communication by sending a request
  • Server — hosts resources and waits for incoming requests; never initiates contact
  • Communication is always request → response, never the other way

● HTTP & TCP

HTTP does not require the transport layer to be connection-based, only reliable. Among TCP and UDP, TCP is more reliable — so HTTP uses TCP. This sits at Layer 7 (Application Layer) of the OSI model; TCP handshakes and TLS encryption live in the layers below.

Evolution of HTTP

Each version of HTTP redefined how clients and servers exchange data, chasing better performance and lower latency.

HTTP / 1.0

Each request opened a new TCP connection. Connection was established and torn down for every single request — very inefficient.

Problem: high overhead per request

HTTP / 1.1

Introduced persistent connections — multiple requests/responses over a single TCP connection. Also added chunked transfer encoding and better caching.

Still the most widely used version today

HTTP / 2.0

Added multiplexing — multiple requests over one connection simultaneously. Uses binary framing, header compression (HPACK), and server push.

Still TCP-based; head-of-line blocking persists
HTTP / 3.0 Built on QUIC — a transport protocol over UDP instead of TCP. Faster connection setup, reduced latency, better packet loss handling, and multiplexing without head-of-line blocking.

HTTP Messages

Every HTTP interaction consists of a request message (sent by the client) and a response message (sent by the server).

● Request Message Structure

  • Method — the action type (GET, POST, PUT, etc.)
  • Resource URL — the path being requested
  • HTTP Version — e.g. HTTP/1.1
  • Host — the domain of the server
  • Headers — key-value metadata about the request
  • Blank line — signals end of headers
  • Body — optional; data the client sends to the server

● Response Message Structure

  • HTTP Version — version used
  • Status Code + Text — e.g. 200 OK
  • Response Headers — metadata about the response
  • Blank line — signals end of headers
  • Response Body — the actual resource or data returned

HTTP Headers

Headers are key-value pairs of metadata sent alongside requests and responses. Think of them like the label written on a parcel — the delivery person reads it without opening the package.

● Types of Headers

  • Request Headers — sent by the client to describe itself and its preferences; e.g. User-Agent, Authorization, Accept
  • General Headers — used in both requests and responses; metadata about the message itself; e.g. Date, Cache-Control, Connection
  • Representation Headers — describe the body being transmitted; e.g. Content-Type, Content-Length, Content-Encoding, ETag
  • Security Headers — control browser behaviour to defend against attacks; e.g. HSTS, Content-Security-Policy, X-Frame-Options, X-Content-Type-Options

● Two Key Ideas

  • Extensibility — headers can be added or customised without changing the underlying protocol; developers can create custom headers like X-Custom-Header
  • Remote Control — headers let the client send instructions to the server; Accept tells the server what format to return, Authorization controls access, Cache-Control controls caching

HTTP Methods

Methods define the intent of a request — not just what resource is being accessed, but what action should be performed on it.

● Core Methods

  • GET — fetch data; must not modify anything on the server
  • POST — create a new resource; has a request body
  • PATCH — partial update; selectively replaces fields in the resource
  • PUT — complete replacement; the request body fully replaces the existing resource
  • DELETE — remove a resource from the server
Rule of Thumb Always use PATCH for updates unless you specifically need a full replacement — that's what PUT is for. Most developers misuse PUT when they mean PATCH.

● Idempotent vs Non-Idempotent

An idempotent method can be called multiple times and always produces the same result.

  • Idempotent — GET, PUT, DELETE
  • Non-Idempotent — POST (calling POST twice creates two resources)

CORS & the OPTIONS Method

Browsers enforce the Same-Origin Policy — by default, a web page can only make requests to its own domain. CORS (Cross-Origin Resource Sharing) is the mechanism that lets servers selectively allow cross-origin requests.

● Simple Request Flow

  • Client sends request; browser automatically adds Origin header
  • Server checks the origin against its CORS policy
  • If allowed, server returns Access-Control-Allow-Origin header with the client's domain or *
  • Browser checks for that header — if present and valid, lets the response through
  • If the header is absent, browser blocks the response and throws a CORS error

● Pre-flight Request Flow

A request becomes a pre-flight request (requiring an extra OPTIONS call first) if any of these conditions are met:

  • Method is not GET, POST, or HEAD — e.g. PUT, DELETE
  • Request includes non-simple headers — e.g. Authorization, custom headers
  • Content-Type is application/json (not form-urlencoded, multipart, or text/plain)
Reality Check Since most apps send JSON with Authorization headers, nearly all real-world requests trigger pre-flight. You'll see OPTIONS requests constantly in your network tab.

● OPTIONS Pre-flight Response Headers

  • Access-Control-Allow-Origin — which origins are allowed
  • Access-Control-Allow-Methods — which HTTP methods are allowed
  • Access-Control-Allow-Headers — which non-simple headers are allowed
  • Access-Control-Max-Age — how long (seconds) the browser can cache this pre-flight response before asking again; reduces repeated OPTIONS calls

Response Status Codes

Status codes communicate the result of a request in a standardised way — without needing to inspect the body or guess from the response shape.

● 2xx — Success

  • 200 OK — request succeeded
  • 201 Created — resource successfully created (POST)
  • 204 No Content — success, but no body to return (common for pre-flight and DELETE)

● 3xx — Redirection

  • 301 Moved Permanently — resource has a new permanent URL
  • 302 Found — temporary redirect
  • 304 Not Modified — cached version is still valid; don't re-download

● 4xx — Client Errors

  • 400 Bad Request — malformed request or invalid input
  • 401 Unauthorized — not authenticated; client must log in
  • 403 Forbidden — authenticated but not authorised
  • 404 Not Found — resource doesn't exist
  • 409 Conflict — state conflict; e.g. duplicate email on registration
  • 422 Unprocessable Entity — semantically invalid input (correct format, wrong values)
  • 429 Too Many Requests — rate limited

● 5xx — Server Errors

  • 500 Internal Server Error — generic unhandled server crash
  • 502 Bad Gateway — upstream server returned invalid response
  • 503 Service Unavailable — server overloaded or down for maintenance

HTTP Caching

Caching reduces unnecessary network round-trips by letting clients reuse previously fetched resources. The server controls caching behaviour via response headers.

● Key Caching Headers

  • Cache-Control: max-age=N — cache this response for N seconds
  • Cache-Control: no-cache — always revalidate with the server before using cache
  • Cache-Control: no-store — never cache; used for sensitive data
  • ETag — a unique fingerprint (hash) of the resource version
  • If-None-Match — client sends its cached ETag; server returns 304 if unchanged, else returns fresh resource
  • Expires — older alternative to max-age; gives an absolute expiry date

Content Negotiation

Content negotiation lets the client tell the server what format it wants, and the server responds accordingly — same endpoint, different representations.

  • Accept — desired response format; e.g. application/json, text/html
  • Accept-Language — preferred language; e.g. en-US
  • Accept-Encoding — supported compression formats; e.g. gzip, br
  • Server picks the best match and reflects it in the Content-Type response header

Advanced HTTP Concepts

● Compression

  • Server compresses the response body before sending it
  • gzip — most widely supported
  • deflate — older, less common
  • Brotli (br) — newer, better compression ratios than gzip
  • Client advertises support via Accept-Encoding; server confirms via Content-Encoding

● Persistent Connections & Keep-Alive

  • HTTP/1.1 default — TCP connection stays open after a response
  • Subsequent requests reuse the same connection — no re-handshake overhead
  • Controlled via Connection: keep-alive or Connection: close header

● Multipart Data & Chunked Transfer

  • Multipart — used for file uploads; bundles multiple data types (file + form fields) in one request body, separated by a boundary string
  • Chunked Transfer Encoding — server sends response in chunks without knowing total size upfront; useful for streaming large responses

SSL / TLS / HTTPS

HTTPS is HTTP with an encryption layer on top. For all practical backend purposes, HTTP and HTTPS follow the same principles — HTTPS just adds security via TLS.

  • TLS (Transport Layer Security) — encrypts data in transit so it can't be read by intermediaries
  • SSL — the older predecessor to TLS; the terms are used interchangeably though SSL is deprecated
  • Certificate — proves the server's identity; issued by a Certificate Authority (CA)
  • HSTS — HTTP Strict Transport Security; forces the browser to always use HTTPS for a domain, preventing protocol downgrade attacks
OSI Note TLS handshakes and certificate validation live at the network layer — they're important context but border into network engineering territory. As a backend dev, you mostly configure this once via Nginx + Certbot and forget about it.
