What is Routing in Backend?

How requests find their handler — static paths, dynamic params, query params, versioning, and catch-alls.

What is Routing?

HTTP methods express the what of a request — your intent, your action. Routing expresses the where — which resource or address on the server you want to perform that action on.

The combination of an HTTP method and a URL path forms a unique key. The server matches this key against its registered routes and executes the corresponding handler.

  • GET /api/books and POST /api/books are two entirely different routes — the method differentiates them
  • The server checks the method first, then the path, concatenates them into a unique route key, and dispatches to the matching handler
Definition Routing is the mapping of a method + URL path combination to a specific server-side handler function.

Static Routes

A static route has a fixed, constant path string — nothing in the path changes between requests. The server always maps this exact string to the same handler.

  • GET /api/books — always fetches all books
  • POST /api/books — always creates a new book
  • The path /api/books is a constant — it never changes regardless of who calls it

Dynamic Routes & Path Parameters

A dynamic route has a variable slot in the path — a placeholder the server fills at runtime with whatever value the client sent. These slots are called path parameters or route parameters.

  • Convention: /api/users/:id — the :id accepts any string value
  • When GET /api/users/123 arrives, the server extracts 123 as the value of id
  • Even numbers are treated as strings inside path parameters
  • This :param convention is universal — Go, Node.js, Python, Java, Rust all use it
Readability GET /api/users/123 reads as: "fetch me data, for users, whose ID is 123." That human-readable semantic expression is the whole point of REST-style routing.

Query Parameters

Query parameters are key-value pairs appended to a URL after a ? symbol. They are not part of route matching — the server uses only the path before the ? to identify the handler. Query params are extra metadata sent alongside the request.

● Why Query Parameters Exist

POST and PUT requests have a body — a container to send user-defined data to the server. GET requests have no body. So when you need to pass extra information with a GET request (like a search term, a page number, or a sort order), query parameters fill that gap.

You could technically put everything in path parameters, but that defeats the semantic purpose of paths. /api/search/some-value loses meaning fast. /api/search?query=some-value is far cleaner.

● Structure

  • Starts with ? after the path
  • Key-value pairs separated by =
  • Multiple params chained with &
  • Example: /api/books?page=2&limit=20&sort=asc

● Common Use Cases

  • Pagination — ?page=2&limit=20; server returns chunk of data for that page
  • Filtering — ?status=active&role=admin
  • Sorting — ?sort=createdAt&order=desc
  • Search — ?query=backend+engineering
Path Params vs Query Params Use path parameters to identify a specific resource. Use query parameters to modify, filter, or add metadata to what you get back.

Nested Routes

Nesting is not a distinct route type — it's a practice you'll see everywhere in REST APIs. It expresses hierarchical relationships between resources through increasingly specific URL paths.

  • /api/users — all users
  • /api/users/:userId — one specific user
  • /api/users/:userId/posts — all posts of that user
  • /api/users/:userId/posts/:postId — one specific post of that user

Each level is its own unique route with its own handler. Stopping at different levels returns different data — /users/123/posts returns all posts, /users/123/posts/456 returns one specific post.

Best Practice Don't go beyond 2–3 levels of nesting. Deep nesting becomes hard to read and maintain. Keep it flat where possible.

Route Versioning & Deprecation

APIs evolve. When you need to introduce breaking changes — changing response shapes, renaming fields, restructuring data — you can't just update the existing route because every existing client will break instantly.

● URI Versioning

Embed the version number directly in the path. Both versions run simultaneously — clients migrate at their own pace.

  • /api/v1/products — original response shape (e.g. { id, name, price })
  • /api/v2/products — new response shape (e.g. { id, title, price })

● Deprecation Workflow

  • Release v2 alongside v1 — nothing breaks on day one
  • Notify all frontend and mobile engineers of the deprecation window
  • Teams migrate their calls from v1 to v2 within that window
  • After the window closes, sunset v1 — redirect or remove it
  • Eventually rename v2 to v1 — the cycle resets

Catch-All Routes

A catch-all route is registered last in the route list. It matches any request that didn't match any defined route — and returns a clean, user-friendly 404 response instead of a null or crash.

  • Pattern: /* — matches everything
  • Without this, unmatched routes return a null response by default — no useful error for the client
  • The catch-all handler returns a proper message: "This route does not exist"
  • Must always be the last registered route — otherwise it swallows legitimate routes
